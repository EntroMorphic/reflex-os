# Nodes of Interest: Phase 26 (LoomScript DSL)

## Node 1: Semantic Weaving
LoomScript must abstract the 9-trit coordinate system. 
*Why it matters:* Developers should think in names (`agency.led`), not spatial vectors.
*The Tension:* Static names in code vs. Dynamic Shadow Paging at runtime. The compiler must produce "Late-Binding" instructions.

## Node 2: Stateful Resonance (The Anti-Loop)
LoomScript replaces `while(1)` with `weave`.
*Example:* `weave source -> sink` implies a continuous, real-time synchronization.
*Innovation:* The code describes the *Topology* of the system, and the Substrate handles the execution.

## Node 3: Atlas-Native Compilation
The compiler must use the SVD/Shadow Atlas as a "Header File."
*Linkage:* If `loomc` knows the Atlas, it can perform compile-time validation: "Error: `agency.invalid.pin` does not exist in the ESP32-C6 territory."

## Node 4: Fragment Packing
A LoomScript program is not a "Process"; it is a **Geometric Fragment**.
*Format:* We need a binary format that packs:
- Required Cells (Names)
- Intent Routes (Sources/Sinks)
- Temporal Transitions (TASM snippets)

## Node 5: The "Live Tapestry" DX
Can we "Hot-Reload" LoomScript?
*The Dream:* Developer changes a line of `.ls` on their PC, the compiler sends a small Arc packet, and the board instantly "Unweaves" the old fragment and "Weaves" the new one without a reboot.
