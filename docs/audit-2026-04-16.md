# Codebase Audit — 2026-04-16

A first-hand, file-by-file assessment of the Reflex OS codebase at commit `6398f38` (post-v2.6.0, post-purpose-routing). Every source file, header, tool, and example was read directly. No subagents.

---

## 1. What Is Genuinely Novel

### 1a. SVD-to-Semantic-Name Catalog Pipeline

**Files:** `tools/goose_scraper.py` → `components/goose/goose_shadow_atlas.c` (9,527 entries) → `goose_shadow_resolve` (binary search) → `goonies_resolve_cell` (on-demand paging)

Nobody else has taken an SVD file, generated a flat sorted catalog of every documented register and field, and made the entire MMIO surface addressable by hierarchical semantic name at runtime on a $5 microcontroller. The two-tier memory model — flash-resident catalog (9,527 entries, read-only) + RTC RAM working set (256 slots, mutable, round-robin eviction) — is a clean, practical solution to a real constraint. `atlas verify` proving the full round-trip on hardware (name, addr, mask, type, coord) across all 9,527 entries is strong validation.

The scraper's ASCII invariant + explicit `encode("ascii")` byte-order sort guarantees that the C-side `strcmp` binary search and the Python-side sort are byte-identical. The reciprocal sync comments between `goose_scraper.py` and `shadow_node_t` in `goose.h` make this a maintainable guarantee rather than an accidental coincidence.

### 1b. Ternary Zero as "Unknown/Undecided"

The project's deepest idea isn't the VM arithmetic — it's using ternary's third value as a first-class representation of uncertainty in a hardware routing substrate. Binary systems collapse unknown state into a default (usually false). This substrate can *hold* it.

The `GOOSE_CELL_PURPOSE` + `weave_sync` + `learn_sync` pipeline is the first concrete expression: the system's routing behavior changes based on what it knows about its own purpose, and the zero state lets routes exist without being committed. This is a genuine contribution to embedded systems design philosophy, not decoration.

### 1c. Distributed Hardware DNS over ESP-NOW

**File:** `components/goose/goose_atmosphere.c`

`goonies_resolve_cell("peer.worker.agency.led")` triggers an `ARC_OP_QUERY` broadcast, peers respond with `ARC_OP_ADVERTISE`, and a ghost cell materializes locally. The atmospheric mesh — HMAC-SHA256 Aura (truncated to 32 bits for wire compatibility), 64-slot time-bounded replay cache, protocol version epoch (`GOOSE_ARC_VERSION=0x02`), NVS-backed key auto-provisioning, and postural consensus with ±10 hysteresis — is implemented and validated on three physical boards. ESP-NOW projects are common. Using ESP-NOW as a naming-service substrate is not.

## 2. What Is Useful and Well-Executed

### 2a. The Ternary VM

**Files:** `vm/ternary.c`, `vm/interpreter.c`, `vm/loader.c`, `vm/mmu.c`, `vm/cache.c`, `vm/syscall.c`, `vm/task_runtime.c`, `tools/tasm.py`

Correct balanced-ternary arithmetic with carry propagation, overflow detection, pack/unpack (2 bits per trit), and self-checks. The interpreter is a clean register machine with 20+ opcodes including GOOSE-native instructions (`TROUTE` for fabric route creation, `TBIAS` for orientation mutation, `TSENSE` for cell state read). The loader does CRC32 verification, 32-bit packed instruction unpacking with 17-bit signed immediates, and full validation (opcode range, register bounds, branch targets, syscall IDs) before execution. The `tasm.py` assembler handles labels, ternary literals (`t+0-`), register operands, and emits the v2 packed binary format.

This isn't a toy — it's a working ternary execution environment with a real toolchain.

### 2b. The Three-State Cache

**File:** `vm/cache.c`

Direct-mapped 16-set cache for `word18` values with I/E/M coherency states (simplified MESI without Shared). Write-back on eviction, `reflex_vm_host_write` for host-side mutation with invalidation. Small, correct, well-tested.

### 2c. Concurrency Discipline

**Files:** `components/goose/goose_runtime.c`, `components/goose/goose_supervisor.c`

`loom_authority` as a single spinlock with cycle-accurate timeout and skip-on-contention semantics. Documented lock ordering (`loom → fabric`). `__atomic_store_n` with release semantics on the route append path. Every hot-path mutation site (`alloc`, `pulse`, `learn_sync`, `weave_sync`, `snapshot save/load`) serialized through the same lock. Comments explain *why* each lock is held, not just that it's held. Non-recursive, with a clear warning on `goose_transition_t` that evolution callbacks must not acquire loom_authority.

### 2d. Boot Robustness

**File:** `main/main.c`, `components/goose/goose_runtime.c`

`RTC_DATA_ATTR fabric_magic` (0xF11BFABE) for cold-boot determinism. Unconditional `loom_authority=0` at init. Boot-count panic detection (threshold=3) with safe-mode fallback. Warm-boot name re-registration (alloc_cell's coord-exists path). 10-second stability marker that resets boot count. Purpose name restore from NVS before supervisor starts.

### 2e. The Aura Security Model

**File:** `components/goose/goose_atmosphere.c`

HMAC-SHA256 via stack-allocated mbedtls context (no heap alloc per packet). Fail-closed with `AURA_ERROR_SENTINEL` on any mbedtls error. Per-board random key auto-provisioning on first boot via `esp_fill_random()`. Protocol version byte for cross-firmware interop. Per-peer version mismatch rate-limiting via 8-slot ring. Posture weight cap (`SWARM_WEIGHT_MAX=4`) enforced on both emit and receive. Known limits documented honestly (32-bit Aura truncation, key extractable from flash/JTAG).

## 3. What Is Understated

### 3a. The Purpose → Routing → Learning → Persistence Loop

The combination of:

1. User declares purpose (`purpose set sensor`)
2. `weave_sync` biases routing toward the purpose domain
3. `learn_sync` amplifies Hebbian reward under purpose (2x)
4. `snapshot_save` persists the learned routes to NVS
5. Reboot restores both purpose and learned state

This is a complete feedback loop from user intent to substrate routing to persistent adaptation. Two boards running the same firmware will genuinely diverge over time based on how they're used. The docs frame this as incremental "Phase 29.5+" work. It's actually the load-bearing demonstration of the project's thesis — awareness as an instrument of purpose, not a goal in itself.

### 3b. The LP Heartbeat

**Files:** `components/goose/ulp/lp_pulse.c`, `components/goose/goose_runtime.c` (lp_heartbeat_init/sync/count)

A parallel RISC-V core ticking at 1 Hz while HP runs FreeRTOS at 160 MHz, with HP mirroring intent state into LP-local globals each supervisor pulse and LP incrementing a counter that HP monitors for liveness. This demonstrates cross-core coherency through shared RTC memory on a heterogeneous dual-core RISC-V system. The limitation (LP can't drive GPIO 15, which is HP-only) is documented honestly rather than hidden.

### 3c. The Scraper as a Reproducibility Tool

`tools/goose_scraper.py` isn't just a code generator — it's a reproducibility guarantee. Given the same SVD file, the scraper produces byte-identical output. The ASCII assertion and explicit byte-key sort ensure that the C binary search and the Python sort agree. The positional-initializer sync comments between the scraper and the header struct make the coupling explicit and maintainable. This kind of tooling discipline is rare in embedded projects.

## 4. What Is Paper-Worthy

### 4a. "Hardware as a Named Geometric Field"

The concrete, publishable claim: take a $5 RISC-V microcontroller (XIAO ESP32-C6), generate a complete semantic name catalog from its SVD (9,527 entries), implement a two-tier paged memory model for those names backed by RTC RAM (256-slot working set with round-robin eviction), add distributed naming across a radio mesh (ESP-NOW with HMAC-SHA256), and show that user-declared purpose can bias autonomous routing — all within the RAM/flash budget of the target.

**Evidence:**
- `atlas verify: ok=9527/9527, duplicates=0, failures=0` (on hardware)
- Three-board mesh trial: 317+ SYNC packets, QUERY→ADVERTISE ghost solidification, postural consensus crossing ±10 hysteresis, HMAC rejection verified
- Purpose persistence across reboot with restored routing bias
- Hebbian plasticity with NVS-backed learned_orientation + hebbian_counter

This is a systems paper. The numbers are real, measured on physical hardware, and unpadded.

### 4b. The Substrate-as-Interface Thesis

The argument that a ternary OS's awareness of its hardware (the Atlas), its user's purpose (GOOSE_CELL_PURPOSE), and its own learned routing preferences (Hebbian plasticity) can make it a better interface — not a smarter computer, but a more useful substrate — is philosophically interesting and technically grounded. The essay at `docs/essay-substrate-as-interface.md` frames this argument well. The code backs it up.

## 5. What Is Fluff

### 5a. goose_dma.c — Scaffold, Not a Feature

**File:** `components/goose/goose_dma.c`

Allocates GDMA TX/RX channels but never transfers data. `goose_dma_apply_route` logs "FLOWING" or "INHIBITED" but has no actual DMA descriptors, no start/stop calls, and no data movement. The ESP_ERROR_CHECK on channel allocation would panic the board if it failed, which is inappropriate for a feature that does nothing. This should be clearly labeled as a stub or removed from the default build.

### 5b. goose_etm.c — Scaffold, Not a Feature

**File:** `components/goose/goose_etm.c`

Maintains a 50-lane ETM pool but never calls `esp_etm_new_channel`, never wires any GPIO events, and never establishes a real hardware connection. `goose_etm_apply_route` sets `in_use=true` in a pool slot and logs "Silicon Route Established" with no actual ETM channel behind it. Same issue as DMA.

### 5c. goose_gateway.c — Largely Dead Code

**File:** `components/goose/goose_gateway.c`

Runs a polling task on `REFLEX_NODE_GATEWAY` looking for `op == 0x10` messages. Nothing in the codebase sends those messages. The gateway concept (bridge legacy fabric messages to GOOSE cells) was sound before the project moved to direct `goonies_resolve_cell` paths, but the implementation is now an artifact.

### 5d. goose_library.c — Half-Implemented with a Crash Bug

**File:** `components/goose/goose_library.c`

`goose_weave_fragment` does the simplest possible thing for HEARTBEAT and GATE (allocate a cell, set a type) and nothing at all for NOT. `goose_weave_loom` has a quota system and route weaving, but `resolved_cells[routes[i].src_idx]->coord` dereferences without a NULL check on the resolved cell — if any `goonies_resolve_cell` call returns NULL, the board panics with a load access fault. The fragment concept is interesting but the code is fragile and under-exercised.

### 5e. bonsai/ — Historical Archive, Not Documentation

60+ markdown files of design journals, experiments, glossaries, morphologies, ontologies, and roadmaps. Some were formative — the LMM journals clearly drove real design decisions. But they're now historical artifacts, not load-bearing documentation. They're not referenced by the current code or docs. A reader coming to the project for the first time would encounter this mass of prose before reaching the actual architecture in `docs/`.

### 5f. journal/ — Same Issue

Phase-by-phase LMM journals at the project root. Useful in the moment, now redundant with `docs/strategy.md` and the git history.

### 5g. lmm_repo/ — Vendored Methodology, Not Project Source

A committed copy of the LMM methodology repo. This belongs as a reference link in the README or CONTRIBUTING.md, not as committed source in the project tree.

## 6. Structural Health

### What's clean

- Module boundaries are clear: `components/goose/` is the substrate, `vm/` is the ternary machine, `core/` is the binary host, `shell/` is the CLI, `services/` is the lifecycle layer, `drivers/` is the HAL.
- Dependency direction is enforced: the VM depends on core only through narrow interfaces, drivers don't depend on the VM, the GOOSE substrate is self-contained as an ESP-IDF component.
- The header files (`goose.h`, `reflex_ternary.h`, `reflex_vm.h`) are well-organized with clear docstrings on public types and functions.
- The build system is simple: one ESP-IDF project with `components/goose/` as a local component, `sdkconfig.defaults` pinning the target, CI via GitHub Actions.

### What could be cleaner

- `goose_runtime.c` is 599 lines and contains the fabric, the naming service, the capability resolver, the weave_sync autonomous fabrication, the LP heartbeat, the purpose API, and the coordinate helpers. It's the project's most modified file and its most load-bearing. A split into `goose_fabric.c` (alloc, eviction, lattice), `goose_goonies.c` (naming, resolution, capability matching), and `goose_runtime.c` (remaining glue) would improve readability without changing any interfaces.
- `shell.c` is ~770 lines and growing. Each major command group (goonies, atlas, mesh, purpose, snapshot, vm) could be its own file with a registration pattern.
- The `bonsai/`, `journal/`, and `lmm_repo/` directories add ~100 files that aren't referenced by the code or docs. Moving them to a separate `archive/` or a dedicated branch would reduce cognitive load for new readers.

## 7. Honest Limits

These are not bugs or regressions. They're the boundaries of what the engineering can currently claim.

1. **Aura wire size**: HMAC-SHA256 truncated to 32 bits. Collision resistance capped at ~2^16 (birthday bound). Protocol epoch lets a future version expand the field.
2. **Aura key storage**: NVS-backed, extractable by anyone with physical flash/JTAG access. Secure-boot + efuse derivation is a tracked follow-up.
3. **LP heartbeat is observation-only**: the LP core can't drive the onboard LED (GPIO 15 is HP-only). It's a parallel counter + intent mirror, not a LED driver.
4. **Catalog ≠ silicon**: `atlas verify` covers 100% of the SVD-documented MMIO surface. Undocumented registers, eFuse bits outside the SVD schema, and silicon-revision deltas are not covered.
5. **Purpose is a singleton**: one name, one cell, one machine. Multi-purpose would require a different architecture.
6. **Hebbian plasticity is simple**: fire-together-wire-together with a fixed threshold (±8). No temporal dynamics, no gradient, no backpropagation. Upgradable without changing the per-route field layout.
7. **No automated tests on host**: all validation is on-device via the serial shell. A host-side test harness (using a mock ESP-IDF or native compilation) would enable CI-level regression testing.

---

## 8. Verdict

The project has a real, novel core: **the SVD-to-semantic-name pipeline, the two-tier paged naming service, the distributed hardware DNS, and the purpose-modulated routing loop.** These are technically sound, hardware-validated, and — taken together — constitute something genuinely new in embedded systems.

The ternary VM is genuine and well-implemented. The concurrency discipline is honest. The boot sequence is robust. The security model is real (not theatrical).

The fluff is concentrated in three areas: scaffold modules that claim functionality they don't have (DMA, ETM, gateway), a NULL-unsafe dereference in the LoomScript loader, and ~100 markdown files in `bonsai/`, `journal/`, and `lmm_repo/` that were formative but are now archival weight.

The biggest risk to the project isn't technical — it's that a first-time reader encounters the journals and scaffolds before they encounter the real thing. The load-bearing work is strong enough to stand on its own. It doesn't need the scaffolding.
